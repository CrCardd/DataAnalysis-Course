O Elasticsearch não usa diretamente o MapReduce da forma tradicional como o Hadoop, mas ele adota um modelo conceitualmente parecido quando executa buscas distribuídas e agregações em um cluster. Vamos entender como isso acontece:

🔍 O que é MapReduce (de forma geral)?

MapReduce é um modelo de programação usado para processar grandes volumes de dados distribuídos:

  • Map: cada nó processa seus dados localmente, gerando pares (chave, valor).

  • Reduce: os resultados locais são combinados para gerar um resultado final.

🧠 Como o Elasticsearch se inspira no MapReduce?

O Elasticsearch funciona em cima de shards distribuídos (partições dos índices), e quando você faz uma busca ou uma agregação, ele usa um modelo similar ao MapReduce:

1. Map (Fase de Consulta Local nos Shards)

  • Quando você faz uma busca ou agregação, o coordenador da requisição envia essa tarefa para todos os shards relevantes.

  • Cada shard executa a busca/agregação localmente (map), retornando seus resultados parciais.

2. Reduce (Fase de Combinação dos Resultados)

  • O coordenador coleta os resultados parciais de cada shard.

  • Em buscas: os documentos são classificados e ordenados.

  • Em agregações: os valores são reduzidos e combinados para formar o resultado final (ex: somar totais, calcular médias, etc).

🧭 Exemplo com agregação

Suponha que você queira saber a média de preços de produtos:

  1. Cada shard calcula a média local de seus próprios documentos.

  2. O nó coordenador coleta essas médias e calcula a média global ponderada, com base no número de documentos em cada shard.

    • Processa localmente os dados e gera pares chave-valor.

  2. Fase Shuffle (Shuffle & Sort):

    • Os resultados dos mappers (shards) são agrupados por chave.

    • Essa fase organiza os dados para que valores com a mesma chave sejam enviados para o mesmo reducer.

  3. Fase Reduce (Reduce Phase):

    • Um nó coordenador (ou reducer) coleta os dados agrupados.

    • Realiza operações finais de agregação, como somar totais ou calcular médias.

    • Retorna o resultado final ao usuário.

🧠 Como o Elasticsearch Implementa Isso

  • Shards como Unidades de Processamento: Cada shard no Elasticsearch pode ser visto como uma unidade de processamento independente, similar a um mapper no Hadoop. Isso permite que o Elasticsearch distribua eficientemente as consultas e agregações entre os nós do cluster.

  • Coletor de Resultados (Reducer): O nó coordenador atua como um reducer, agregando os resultados parciais dos shards para produzir o resultado final.

  • Uso do Elasticsearch-Hadoop (ES-Hadoop): Para integrar o Elasticsearch com o Hadoop, o ES-Hadoop fornece implementações de InputFormat e OutputFormat que permitem ler e escrever dados entre o Hadoop e o Elasticsearch. Isso facilita a execução de tarefas MapReduce que envolvem dados armazenados no Elasticsearch.

Boa pergunta! Vamos explicar de forma simples o que são TF-IDF e BM25, que são duas fórmulas usadas pelo Elasticsearch (e por motores de busca como o Google) para calcular a relevância de documentos em uma busca.

🔍 O que são TF-IDF e BM25?

🧠 Objetivo:

Determinar quais documentos são mais relevantes para uma palavra ou frase buscada.

1. TF-IDF (Term Frequency – Inverse Document Frequency)

É uma fórmula clássica que mede:

"Quão importante é um termo em um documento, considerando toda a coleção de documentos."

✅ Explicado em partes:

  • TF (Frequência do termo):
    Quantas vezes a palavra aparece no documento.
    → Quanto mais aparece, mais importante pode ser.

  • IDF (Frequência inversa do documento):
    Penaliza palavras muito comuns (como "o", "e", "de").
    → Palavras que aparecem em muitos documentos são menos informativas.

🧮 Fórmula (simplificada):

ini
TF-IDF = TF * log(N / DF)

  • TF = frequência do termo no documento.

  • N = número total de documentos.

  • DF = número de documentos que contêm o termo.

📌 Exemplo real:

  • Palavra "energia":

    • Documento A: aparece 5 vezes (TF = 5)

    • "energia" aparece em 100 de 1000 documentos (IDF = log(1000/100) = 1)

    👉 Relevância (TF-IDF) = 5 × 1 = 5

  • Palavra "o":

    • Aparece 20 vezes (TF = 20)

    • Mas está em todos os documentos (IDF = log(1000/1000) = 0)

    👉 Relevância (TF-IDF) = 20 × 0 = 0

2. BM25 (Melhorador do TF-IDF)

BM25 é uma fórmula mais moderna e eficaz que melhora o TF-IDF, levando em conta:

  • Comprimento do documento (para não favorecer textos gigantes).

  • Frequência de termos com limite (não adianta repetir a palavra 1000 vezes).

✅ Por que BM25 é melhor?

  • Penaliza exageros: repetir uma palavra demais não aumenta muito a pontuação.

  • Compensa tamanho: documentos muito longos não são favorecidos automaticamente.

  • Elasticsearch usa BM25 como padrão desde a versão 5.0.

🔄 O que é DAG?

DAG significa:

Directed Acyclic Graph
(Grafo Acíclico Direcionado)

É uma estrutura de dados gráfica usada para representar dependências entre tarefas ou etapas de um processo. No contexto de Big Data, é usada para modelar e otimizar fluxos de processamento de dados.

🧠 Conceito simplificado

  • Nódulos (ou vértices) = tarefas a serem executadas.

  • Setas (arestas direcionadas) = uma tarefa depende da anterior para começar.

  • Acíclico = não há ciclos (ou seja, uma tarefa não depende de si mesma, direta ou indiretamente).

📦 Exemplo real com DAG (Spark)

Imagine que você quer processar dados assim:

  1. Ler dados de um CSV.

  2. Filtrar linhas com erros.

  3. Calcular a média de uma coluna.

  4. Gravar os resultados em um banco.

Isso vira um DAG:

nginx
Leitura ──► Filtro ──► Média ──► Escrita

Cada etapa depende da anterior. E como não há voltas, é um grafo acíclico.

✅ Por que DAG é importante?

1. Otimização de execução

  • Sistemas como Apache Spark, Airflow ou Dask usam DAGs para saber qual tarefa pode rodar em paralelo e em qual ordem.

2. Falhas controladas

  • Se uma etapa falha, o DAG ajuda a saber quais etapas precisam ser reexecutadas.

3. Planejamento automático

  • No Spark, você escreve o código sequencialmente, mas ele monta o DAG por baixo dos panos e o executa da melhor forma possível.

🔄 Modelo geral sugerido (como você propôs):

1. Divisão de tarefas (Map/Scatter/Distribute)

  • Dados ou operações são quebrados e enviados a múltiplos nós (shards, partições, workers, etc.).

  • Cada nó executa um trabalho independente com sua fatia de dados.

2. Processamento paralelo

  • Tarefas são executadas em paralelo (às vezes em pipelines, às vezes em estágios).

  • Pode envolver filtros, joins, transformações, análises, etc.

3. Redução, agregação ou sincronização (Reduce/Gather/Join)

  • Os resultados locais são combinados.

  • Pode ser um resultado final global (como uma soma), uma ordenação global, ou apenas sincronização entre nós.

Esse modelo pode ser representado por um DAG acíclico, onde:

  • Vértices = tarefas.

  • Arestas = dependências entre etapas.

🧠 Por que isso faz sentido didaticamente?

  • Ajuda a ensinar alunos e profissionais a ver padrões recorrentes em Spark, Hadoop, Flink, Dask, Elasticsearch, etc.

  • Fornece uma visão mental clara: “Distribua → Processe → Reúna”.

  • É compatível com a abstração funcional usada em MapReduce, Spark (map, reduceByKey, collect), Pandas (groupby().agg()), etc.

A chave para garantir que Big Data realmente agregue desempenho está em entender e controlar as condições que favorecem a distribuição sem prejudicar a performance. Vamos detalhar essas condições e como elas influenciam a arquitetura e as escolhas tecnológicas.

💡 Condições para que Big Data realmente agregue desempenho

1. Baixo custo de comunicação entre os nós

  • Comunicação é o inimigo silencioso. Em qualquer sistema distribuído, a comunicação entre os nós (e especialmente entre as partições de dados) pode ser muito custosa em termos de tempo e recursos.

    • Custo alto: Transferir grandes volumes de dados pela rede pode ser muito mais lento do que realizar cálculos locais, além de causar congestionamentos e aumentar a latência.

    • Solução: Minimizar a necessidade de comunicação entre os nós.

      • Por exemplo, sistemas como Apache Spark e Flink buscam minimizar o embaralhamento de dados (shuffle) — que é quando os dados precisam ser redistribuídos entre nós para realizar operações de agregação ou join.

      • Ajustes de localidade de dados: Certificar-se de que dados relevantes estejam em nós próximos (ex: data locality no Hadoop ou Spark), de modo a reduzir o tráfego de rede.

2. Granularidade de divisão de dados

  • Divisão muito fina ou muito grosseira pode afetar o desempenho.

    • Divisão muito fina: Pode resultar em overhead de gerenciamento muito alto. Isso acontece quando você tenta paralelizar demais, mas acaba gastando mais tempo com a gestão das tarefas pequenas do que realmente processando os dados.

    • Divisão muito grosseira: Também pode resultar em subutilização dos recursos. Se a partição for grande demais, você acaba não aproveitando o paralelismo de forma eficiente.

    Solução: Encontrar um tamanho ideal de partição. Muitos frameworks oferecem configurações para particionamento de dados (como o número de partições no Spark ou no Elasticsearch), e é importante ajustar isso conforme a carga e a arquitetura de hardware.

3. Eficiência no uso de recursos (CPU, memória, disco)

  • Desbalanceamento de carga pode ocorrer quando alguns nós estão sobrecarregados enquanto outros estão ociosos. Em um sistema de Big Data, isso pode ser causado por:

    • Dados desbalanceados.

    • Algoritmos ineficientes que fazem com que certos nós executem mais trabalho.

    Solução: Sistemas como Apache Spark usam balanceamento dinâmico de carga. Além disso, algoritmos eficientes de divisão de dados e processamentos minimizam a sobrecarga.

  • Uso de memória e armazenamento: Em muitos sistemas distribuídos, se os dados não cabem na memória, isso pode levar a operações de leitura/escrita excessivas em disco, o que é muito mais lento.

    Solução: Manter dados em memória sempre que possível (ex: o uso de in-memory computing em Spark).

4. Minimizar operações de agregação e joins caros

  • Shuffling de dados (quando dados precisam ser redistribuídos para outras partições durante operações como join ou groupby) é uma das operações mais caras em Big Data, pois exige grande comunicação entre os nós e reestruturação de dados.

    Solução:

    • Evitar shuffle desnecessário: Em muitos casos, pode-se usar agregações locais em cada nó antes de realizar operações de redução global.

    • Optar por algoritmos de join eficientes (por exemplo, usar joins particionados no Spark).

5. Paralelismo adequado ao hardware

  • Em sistemas distribuídos, não adianta nada distribuir tarefas se o hardware de cada nó não for capaz de processar essas tarefas de forma eficiente. A latência de rede e o uso inadequado de CPU/memória podem anular qualquer ganho de desempenho obtido pela distribuição.

    Solução: Ajuste o número de núcleos e memória disponível em cada nó, e certifique-se de que o sistema está utilizando ao máximo os recursos locais.

🔄 Exemplo prático de perdas e ganhos de desempenho

Cenário: Processamento de logs em tempo real

  • Problema: Processar bilhões de logs em tempo real (por exemplo, de servidores web ou sensores IoT) e gerar métricas de performance (como número de acessos por IP ou tempo médio de resposta).

  • Solução tradicional: Uma máquina só. Isso vai resultar em alta latência (demora para processar todos os dados) e bottle neck no disco ou memória.

  • Solução distribuída: Distribuir o processamento em múltiplos nós. Cada nó processa uma parte dos logs, reduzindo o tempo total.

Desafio de desempenho:

  • Custo de comunicação: Se, ao final do processo, você tiver que agregar os dados de todos os nós em uma única máquina para calcular as estatísticas finais, você poderá ter uma sobrecarga significativa na comunicação.

  • O sistema precisa de algoritmos de agregação eficientes para evitar o tráfego de dados desnecessário.

Como evitar perda de desempenho:

  • Processamento em paralelo: Cada nó pode processar logs em tempo real de uma parte dos servidores ou sensores.

  • Agregação local: Realize agregações parciais em cada nó (ex: contar acessos por IP em cada nó), e apenas envie as contagens totais para um nó central para agregação final.

📈 Conclusão: Como evitar perda de desempenho

  1. Minimizar comunicação: Reduza a troca de dados entre nós sempre que possível, preferindo agregações locais antes de realizar reduções globais.

  2. Balanceamento de carga: Divida os dados de forma eficiente para evitar sobrecarga de certos nós.

  3. Evitar operações caras de agregação e shuffle: Utilize algoritmos eficientes para joins e agregações.

  4. Ajustar recursos: Otimize o uso de CPU, memória e armazenamento em cada nó para que o sistema como um todo seja eficiente.

  5. Leve em conta latência de rede e custo de comunicação: Projete a arquitetura para que os dados que precisam ser processados juntos, fiquem o mais próximo possível, minimizando o tráfego de rede.
